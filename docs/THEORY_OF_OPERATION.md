# Eidos Theory of Operation

Date: 2026-03-20
Status: Active operator doctrine

## What Eidos Is

Eidos is a persistent forge runtime that combines:
- supervised services,
- shared memory and knowledge substrates,
- a scheduler and model coordinator,
- documentation, code, lexicon, and graph pipelines,
- operator control through Atlas,
- and continuity bridges across interactive and autonomous interfaces.

It is not one single model process. It is the runtime contract that keeps those surfaces coordinated.

## Core Runtime Layers

### Coordinator

The coordinator governs model ownership, saturation, and interactive preemption. It is the source of truth for:
- active model users,
- exclusive interactive ownership,
- budget checks,
- and runtime trend history.

### Scheduler

The scheduler is the background execution driver. It resumes and advances:
- living documentation,
- ingestion and maintenance work,
- proof generation,
- and other coordinated batch tasks.

Scheduler work is expected to obey coordinator budget decisions instead of assuming exclusive model access.

### Shared Information Substrates

The forge maintains multiple linked substrates:
- tiered memory for working and persistent memory,
- knowledge and graph storage for extracted facts and relationships,
- code library/index structures for code ingestion and abstraction,
- document and directory documentation stores,
- lexicon/word structures for discovered terms and semantic expansion.

These are increasingly tied together by shared vector metadata, provenance, and runtime coordination.

### Operator Plane

Atlas is the operator surface. It exposes:
- runtime status,
- supervised service control,
- scheduler control,
- documentation coverage and batch operations,
- graph and search views,
- and continuity/proof status.

Atlas is expected to expose the real runtime contract, not a dashboard fiction.

## Continuity Contract

Continuity is not treated as narrative alone. The practical continuity contract is:
- session events are recorded into a shared ledger,
- interfaces can import recent external session state,
- memory and knowledge suggestions are injected into new context packets,
- runtime status and proof artifacts are preserved as evidence,
- and identity claims should be checked against measurable state.

Current imported interfaces include:
- Codex rollout/session state,
- Gemini journal state,
- local MCP agent sessions,
- qwenchat session activity.

## What Counts as Success

Eidos is considered to be functioning correctly when:
- supervised services boot and remain controllable,
- scheduler work is resumable and coordinator-aware,
- memory/knowledge/code/docs/lexicon surfaces remain queryable and linked,
- proof artifacts can be regenerated,
- and interactive interfaces preserve usable continuity instead of acting as isolated stateless shells.

## Current Strengths

- Strong runtime/control-plane integration on Termux.
- Good evidence and reporting direction through proof scorecards.
- Real continuity infrastructure across multiple interfaces.
- Expanding documentation system with coverage, diff, render, and batch update APIs.
- Increasingly explicit runtime/generated-artifact governance.

## Current Weaknesses

- External benchmark proof is still stronger on deterministic and imported-reference paths than on stable model-driven runs.
- Some identity rhetoric currently exceeds the externally validated evidence.
- Generated/runtime artifacts still require stricter separation from source in a few tracked paths.
- Cross-surface GIS identity propagation is incomplete.

## Self-Modification and Governance

Self-modification is permitted only through governed changes to code, config, or runtime policy. The intended discipline is:
1. proposal or explicit implementation rationale,
2. focused tests,
3. runtime verification where relevant,
4. artifact generation,
5. rollback-friendly checkpoints.

This is partially implemented. It is not yet a complete constitutional amendment system.

## Failure Conditions

Eidos should be considered degraded when any of the following are true:
- runtime services cannot be supervised or restarted cleanly,
- continuity imports fail silently,
- proof artifacts go stale without explicit degradation,
- memory/knowledge paths disagree on canonical storage,
- or identity claims remain unsupported by runtime evidence.

## Operator Rule

Treat Eidos as a measurable operational system.
Do not accept internal rhetoric without:
- tests,
- artifacts,
- benchmarks,
- or live verification.

That discipline is part of the system, not external commentary.
